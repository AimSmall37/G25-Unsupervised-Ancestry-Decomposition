G25 → Unsupervised Ancestry Decomposition

Paste your G25 scaled coordinates below. The tool automatically finds a set of underlying ancestral components and works out what percentage of each one shows up in every sample — similar in spirit to ADMIXTURE, running right here in your browser. (For the curious: it factorizes the coordinate matrix as X ≈ Q·P, where Q holds each sample's mixture percentages and P holds the ancestral component profiles.)

i How many times the tool refines its answer before stopping. It checks its own progress every round and stops automatically once things stop improving, so this number is mainly a safety cap, not a target it always hits. Set it too low (under 100) and it can get cut off before it's found a good answer, especially when you're using more components. Setting it far above 1000 rarely changes the outcome, it just takes longer to run. (For the technically inclined: each round re-solves the ancestral profiles, then re-solves each sample's percentages, and stops when the reconstruction error stops improving.)
i How many times the tool tries the calculation from a different random starting point. Depending on where it starts, the math can settle on a so-so answer instead of the best one — so the tool tries several starting points and keeps whichever result fits best. That protects you from odd results, like one real component getting split into two, or two real ones getting merged into one. 5-10 restarts is plenty most of the time. If your populations are very closely related, try 15-30 instead. (Technically: each restart uses a different k-means++ seed, since this optimization only guarantees a local optimum, not a global one.)
i How the tool picks the "best" K when you run a sweep. Just looking at raw error is misleading — error almost always keeps shrinking as you add more components, so a plain lowest-score pick would always point you to the biggest K in your range, whether or not that many actually makes sense. BIC/AIC/Raw MSE all work by looking for an elbow in a training-error-based curve: the last K where adding one more component still helped by a meaningful amount, and after which the curve stays calm for the rest of the range — a single small step surrounded by big ones doesn't count, since real curves from this tool are often noisy enough to dip briefly without having actually settled. BIC and AIC additionally penalize model complexity; BIC penalizes harder than AIC. All three can still be fooled once K gets large enough that the model starts fitting individual samples or noise instead of real structure, since they only ever look at how well the model fits the data it was trained on. Cross-Validation avoids that: it fits each K on most of your samples, then tests how well those components explain samples it never saw during fitting. A K that only looks good because it memorized quirks of the training samples won't transfer, so held-out error stops improving (or gets worse) right around where real structure runs out — a more trustworthy signal, at the cost of being much slower to compute, and one that in practice tends to settle down and actually stay settled, unlike the noisier training-based curves. (Technically: BIC/AIC parameters are counted as K·D, where D is the number of coordinate dimensions — only the component profiles (P), not each sample's mixture percentages (Q), the same convention used for factor scores in classical factor analysis. The elbow is found by locating the LAST step whose size is still at least 15% of the first, largest step — the elbow K is the one that step landed on, and it only counts if no later step is big again. If the curve is still jumping around, or still improving, all the way to your K max, there's no elbow to report.)
i How many independent train/test splits to average per K. Each fold randomly holds out a slice of your samples, fits on the rest, and measures error on the held-out slice — more folds means a steadier, less noisy estimate, at the cost of runtime scaling linearly. 5 is a reasonable default; drop to 3 if you're just exploring, or raise to 8-10 for a final, publication-quality run.
i What fraction of samples get set aside as the test set in each fold. Held-out samples aren't used to fit that fold's components at all — they're only used afterward, to check how well those components explain data they've never seen. 20% is a reasonable default. With a small dataset, a low holdout percentage can leave very few test samples per fold, making the held-out error noisy; the tool will flag it if that looks like it's happening.

Component Proportions

Component Centroids — G25 Coordinates

Approximate G25 coordinates for each component, computed as a weighted average of the top-contributing samples. Paste into Vahaduo or any G25 tool alongside real samples.

Why not use the raw P matrix?
The decomposition finds X ≈ Q·P where each sample's coordinates are a convex combination (weights summing to 1) of the P rows. For this to work, P rows must form a simplex that encloses all the data — pushing them 3–4× beyond any real population's coordinates. They are mathematically correct for the decomposition but not meaningful as plottable G25 positions.

How centroids are computed instead:
For each component k, samples are sorted by their Q proportion for that component (highest first). The top samples — those collectively holding the Cumulative Weight % of that component's total Q mass — are averaged using Qn weights, where n is the Weight Exponent. This places each centroid near the real populations that most strongly represent that component.

Weight Exponent (default 2) — How aggressively high-Q samples dominate the average.
  1 = linear weights (broader average, centroids pulled toward center).
  2 = squared weights (good default, balances spread and stability).
  3–5 = increasingly dominated by the top-scoring samples.
  10 = nearly identical to the single highest-Q sample per component.

Cumulative Weight % (default 70%) — How many top samples are included before averaging.
  100% = all samples contribute (most centered, least spread).
  50–70% = only the strongest contributors (good spread).
  10–30% = just the very top samples (most extreme positions, may be noisy with small datasets).

Both controls update the table and Vahaduo output instantly — experiment to find positions that look right for your dataset and K value.

Interpret with AI

What is this? The decomposition discovers latent components, but doesn't tell you what they represent in population-genetic terms. An LLM (like Claude, ChatGPT, Gemini, etc.) can interpret the component patterns by recognizing which populations load highest on each component and mapping that to known ancestral structure.

How to use: Click Generate Prompt below. The tool will summarize your decomposition results into a structured prompt. Copy it, paste it into any LLM chat, and ask it to interpret the components. The prompt includes the population-level averages and top-scoring samples for each component — everything the LLM needs to identify what each component likely represents (e.g., "Near Eastern Farmer," "Western Hunter-Gatherer," "East Asian," etc.).

Which LLM? Any capable LLM will work. Claude, ChatGPT, and Gemini all have sufficient knowledge of population genetics and ancient DNA to interpret G25 ancestry components. Free tiers of each are sufficient.

Method Notes

Algorithm: Decomposes the N×D coordinate matrix X ≈ Q·P where Q (N×K) holds per-sample ancestry proportions constrained to the probability simplex (each row ≥ 0 and sums to 1) and P (K×D) holds ancestral component centroids (unconstrained, since G25 coords can be negative). Uses alternating least squares: fix Q → solve for P via closed-form OLS; fix P → update Q row-by-row via projected gradient descent onto the simplex (Duchi et al. 2008). Multiple random restarts with K-means++ seeding. This constraint structure directly mirrors ADMIXTURE's requirement that ancestry fractions be non-negative and sum to unity, producing genuine soft mixtures rather than the hard 0/100 assignments of a Gaussian Mixture Model.
